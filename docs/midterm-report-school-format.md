## 二、已完成的研究工作及成果

### （1）系统架构与基础能力

项目已完成前后端基础架构搭建与联调。后端已形成研究任务统一接口与任务对象管理机制，支持：
- 创建任务：`POST /api/research`
- 查询状态：`GET /api/research/:id`
- 获取产物：`GET /api/research/:id/artifact/:name`

任务对象已具备完整生命周期管理，状态覆盖 `created/planning/collecting/reading/writing/reviewing/done/failed`，并支持关键产物输出（`plan.json`、`sources.json`、`notes.json`、`report.md`、`critic_report.json`）。

### （2）端到端自动化研究流程

已实现“主题输入→计划生成→搜索抓取→证据提炼→报告写作→质量检查”的闭环流程：

1. **Planner 规划**：将主题拆分为子问题与关键词；  
2. **Researcher 检索**：调用 Tavily 获取结构化搜索结果（title/url/snippet/source）；  
3. **Fetcher 抓取**：尝试抓取网页正文并清洗；对反爬或抓取失败场景自动降级为 snippet 证据；  
4. **Reader 提炼**：生成 claim/support/url 形式的证据要点；  
5. **Writer 写作**：基于证据要点生成小节正文并附来源链接；  
6. **Critic v0 检查**：检查来源数量、域名多样性与硬断言风险，并执行有限自动修正。

### （3）可解释性与工程鲁棒性

已形成较完整的可解释执行链路（trace），每个阶段均记录 `reasoning/action/observation/decision/final` 事件，且前端支持展开查看 payload 细节。当前 trace 可展示：搜索 query、搜索结果预览、provider 标识、抓取 URL、抓取失败原因、重试原因与等待时长、降级策略触发等。

针对实际运行中的不稳定问题（如模型过载、JSON 解析失败、站点反爬），已完成下列保障：
- LLM 调用自动重试（指数退避）；
- Reader/Writer 解析失败自动降级，不再“一点失败全任务失败”；
- 抓取失败自动回退 snippet 证据；
- 服务端加入优雅关闭，降低热重载端口冲突导致的服务抖动。

## 四、存在的问题与困难

当前项目推进中仍面临以下问题：

1. **外部依赖不稳定**：LLM 端偶发过载、超时或输出不规范 JSON，虽有重试与降级，但会增加执行时长；  
2. **网络抓取限制明显**：部分站点反爬强，导致正文抓取失败率较高，需要依赖 snippet 兜底；  
3. **来源质量参差不齐**：搜索结果中仍可能出现低质量站点，需要进一步加强信源评估与过滤策略；  
4. **质量评估仍偏基础**：当前 Critic v0 能做最小检查，但尚未实现严格的多源事实交叉验证；  
5. **任务持久化不足**：当前内存存储模式在服务重启后可能丢失任务上下文，不利于长期复现实验。

针对以上问题，已在系统中加入“自动重试+失败降级+trace 透明化”机制，并在后续计划中明确了持久化、强评测与强验证的迭代路径。
