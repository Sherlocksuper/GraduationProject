第3章  系统设计

3.1  引言

本章在需求分析基础上完成系统设计，明确总体架构、功能模块划分、核心实体与数据存储结构，并对认证与会话、智能对话与工具编排、知识库与检索增强、深度研究管线及过程展示等关键功能给出设计说明。最后补充错误处理、并发与界面等非功能性设计要点，为第4章系统实现与第5章测试提供依据。

3.2  系统架构

本系统为 B/S 与前后端分离：前端采用 React+Vite 单页应用，经 JSON 调用 Express（Node.js）后端；会话与消息、知识库、研究任务等持久化于 SQLite（默认 data/app.db），用户账户等当前存于 data/users.json，逻辑上仍归为用户域。大模型、嵌入与联网检索由环境变量配置的外部服务提供，经封装客户端调用，与业务进程解耦。总体结构见图3-1。

前端采用 React 组织页面与组件，负责路由、会话与聊天等交互状态；Vite 负责开发与打包。用户在浏览器中操作界面，由前端将登录、会话、知识库、研究等请求以 HTTP（开发环境可为本地 HTTP）与 JSON 载荷发送至服务端接口。

服务端基于 Express 注册路由与中间件：除注册、登录、邮箱验证等公开接口外，受保护接口在统一鉴权逻辑中解析 Bearer Token，未携带或无效时返回 401；通过后进入各业务路由处理函数，调用聊天仓储、知识库仓储、研究存储及大模型客户端等模块完成业务并返回 JSON。

数据方面，会话、消息、知识库分块与研究任务等结构化数据存放在 SQLite（app.db）；用户账户、验证与验证码等当前实现为本地 JSON 文件（如 users.json），与库表在部署上分离但在领域上仍视为用户数据。大模型推理、向量嵌入与联网检索由外部服务承担，服务端仅保存调用所需配置与业务侧产物，不在库内持久化模型参数。

前后端职责清晰、接口以 REST 为主，便于替换模型供应商、扩展新路由或独立演进前端展示，有利于后期维护与功能扩展。

工具编排（ReAct）与检索增强对话建立在会话之上：前者在需要确定性能力时由模型发起工具调用并由服务端回填观察；后者在会话绑定知识库后，对问句做嵌入、多库检索与片段注入，再参与生成。

本系统不设独立管理端，运维类能力可在后续版本中扩展。

3.4.2  关键实体类解释

用户（User）：系统访问主体，保存登录凭据、邮箱验证状态、密码重置与登录验证码等字段；登录成功后由服务端签发令牌供后续请求携带。

聊天会话（ChatSession）：用户在会话维度组织多轮对话，保存标题、创建与更新时间，以及本会话绑定的知识库集合标识列表（JSON 数组形式持久化）。

聊天消息（ChatMessage）：隶属于会话，记录角色（用户或助手等）、正文、时间戳及可选元数据（如关联的研究任务标识），用于前端渲染与上下文组装。

知识库集合（RagCollection）与分块（RagChunk）：集合描述知识库名称及所用嵌入模型标识；分块存储被切分后的文本与对应向量（以 JSON 形式存储于 SQLite 字段中，便于实现阶段简化序列化）。

研究任务（ResearchTask）：描述一次深度研究运行的主题、状态、错误信息、结构化产物（如规划、笔记、报告等 JSON 对象）及轨迹数组，供轮询接口与前端展示使用。

索引设计：在 chat_sessions 上建立 (username, updated_at DESC) 索引以加速会话列表查询；在 chat_messages 上建立 (session_id, id) 索引以加速按会话拉取消息顺序；在 rag_chunks 上建立 (collection_id, chunk_index) 索引以支持分块遍历与检索实现；在 research_tasks 上建立 (username, updated_at DESC) 索引以支持任务列表与清理策略。

用户数据文件：当前实现将注册用户条目存放于 data/users.json 的根对象 users 下，键为用户名，值为包含密码、邮箱、验证令牌与过期时间、登录一次性验证码等字段的对象。论文中可将其视为「用户逻辑表」，若后续改为关系库存储，可保持字段语义不变迁移至 user 物理表。

3.6.3  知识库管理与 RAG 增强对话功能

知识库侧提供集合增删、分块写入与更新、批量删除等接口；嵌入由独立模块调用外部嵌入 API 生成向量并写入 embedding_json。对话侧在用户绑定集合后，将用户问题嵌入为查询向量，对各集合分别检索后按相似度合并截断，将片段格式化写入系统提示或附加消息，再调用大模型生成回答；可配置最低相似度阈值，低于阈值时整轮不注入片段以避免噪声。

3.6.4  深度研究（规划—检索/抓取—证据提炼—写作—评审）功能

深度研究任务由独立存储记录状态与产物；运行管线在服务端异步或同步推进（实现上可与 HTTP 请求生命周期绑定），阶段一般包括：规划子问题、联网检索与网页文本获取、阅读器抽取证据要点、写作器生成章节化报告、批评者进行基础检查与可选重写、全局润色（若启用）。各阶段向 trace 数组追加事件，向 artifacts 字典写入命名产物，供前端轮询展示。

3.7  系统非功能性设计及描述

3.7.1  LLM 与外部接口错误处理与重试

对大模型聊天、嵌入及外部检索等调用设置超时与重试策略；对模型输出 JSON 解析失败时进行有限次重试或降级为纯文本段落，避免整条研究任务因单次解析失败而不可用。HTTP 层统一返回结构化错误码与中文提示信息，前端对 401、409 等状态做分支提示。

3.7.2  并发与资源控制

对同一会话并发的深度研究请求在服务端以映射表防重入，避免同一用户界面重复提交导致多任务写乱；研究任务数量可按用户维度设置上限并淘汰最早任务。消息表对单会话消息条数可截断保留最近若干条，以控制上下文规模与存储膨胀。

3.7.3  用户界面与交互设计

聊天区采用清晰的消息气泡与时间顺序；代码与列表类内容可结合 Markdown 渲染；知识库管理页提供集合与分块列表及操作按钮；深度研究页突出任务状态、耗时预期与 trace 可读性。具体配色与布局以实现为准。
