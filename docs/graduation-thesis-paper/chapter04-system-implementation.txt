第4章  系统实现

4.1  引言

本章围绕第3章设计，对系统实现情况进行说明。实现采用 npm 工作区划分服务端与前端两个子工程：服务端基于 Node.js 与 Express 提供 HTTP 接口与业务编排；前端基于 React 与 Vite 构建单页应用。持久化方面，聊天会话与消息、知识库及研究任务等写入本地 SQLite 数据库；用户注册信息等以 JSON 文件形式落盘，由用户仓储层读写。以下按模块介绍关键实现点，并在相应位置给出截图占位与必要的 PlantUML 出图源码（渲染后插入排版稿；分隔线及说明勿粘入 Word 正文）。

4.2  认证与用户模块

4.2.1  注册、登录与鉴权

认证与鉴权在服务端统一注册为若干 REST 接口：注册时写入用户名、密码与邮箱，并向注册邮箱发送 6 位数字验证码；用户在界面输入验证码后由服务端校验，通过后将邮箱标记为已验证。登录支持「用户名 + 密码」与「邮箱 + 验证码」两种方式；忘记密码时向邮箱发送重置链接，用户凭链接在页面设置新密码。受保护接口在请求头中携带令牌，由中间件解析身份；前端启动时通过「当前用户」类接口恢复登录态，登出时丢弃本地令牌即可。

4.2.2  个人会话与消息持久化

会话与消息由服务端仓储层封装，对应数据库中的会话表与消息表。接口层提供会话列表、新建会话、按会话拉取消息、删除会话以及更新会话标题或「绑定知识库」等能力。追加消息时同步维护会话的最近更新时间；在用户发出首条用户消息时可自动生成会话标题摘要，并对单会话消息总量做上限控制，仅保留最近若干条以保证性能。

4.3  智能对话模块

4.3.1  对话接口与上下文组装

核心入口为已登录用户调用的「发送消息」接口（POST /api/chat）。请求体以当前会话编号与用户问题为主，并可附带对话风格、是否强制走深度研究、知识库检索偏好及简单问答时的生成长度等可选参数。服务端先确认会话归属并读取近期对话作为上下文，再综合用户选择与规则判断本次是否需要「深度研究」：一般闲聊等无检索必要的问题走简单问答，否则进入多阶段研究流程。

未走深度研究时，若该会话绑定了知识库，则先做向量检索并按阈值筛选命中片段，将检索结果以附录形式并入本轮提示；随后将系统说明、历史轮次与当前问题一并送入大模型生成回复，写入会话消息表并向前端返回本轮结果及检索统计信息，便于界面展示是否命中知识库等。

走深度研究时，将上文与当前问题整理为研究主题（可同样附带知识库检索摘要），在数据库中登记研究任务并记录协调过程；研究管线在后台异步执行，完成后把生成的报告或错误说明作为一条助手消息写入当前会话，前端通过任务编号轮询进度直至结束。

（截图占位：图4-6 普通对话一轮问答后的聊天区截图（可见用户与助手各一条）。）

4.3.2  ReAct 编排与内置工具注册实现

工程在服务端单独实现了一套与主对话接口解耦的 ReAct 编排子系统：由编排器按迭代上限循环调用大模型，解析其返回的工具调用与终稿字段，再交由工具执行层完成计算器、时间、回显等内置能力的调用，并通过注册表管理可扩展工具。该子系统体现第3章中的 ReAct 范式，可作为「工具增强对话」的可复用组件；当前主会话仍以直连大模型与深度研究管线为主，二者在架构上预留了后续对接或合并的余地。

（若论文答辩需演示工具循环，可在此补充截图占位：图4-8 调用该子系统的调试脚本或未来接入后的对话效果截图。）

4.4  知识库与 RAG 模块

4.4.1  文档分块与入库

知识库能力通过服务端一组 REST 接口对外提供：支持创建集合、按条或批量写入文本分块等。上传或录入的正文先按固定策略切分为片段，再调用嵌入接口生成向量，与原文一并写入知识库分块表，供后续检索使用。

（截图占位：图4-9 知识库「新建集合」或集合详情页截图。）

（截图占位：图4-10 分块列表或上传/ingest 操作成功后的界面截图。）

4.4.2  向量检索与多库合并、阈值过滤

单库可按向量相似度查询；对话侧若绑定了多个知识库，则分别检索后在服务端合并结果，按相似度排序并截断为前若干条，再按用户或默认设定的最低分门槛整体过滤：若最高分仍低于门槛，则本轮不向模型注入知识库片段，避免噪声干扰。

4.4.3  对话中 RAG 片段注入与引用策略

命中片段在服务端格式化为带编号与相似度说明的附录文本，拼入系统侧提示，引导模型在确有依据处使用统一形式的引用标记。简单问答与深度研究共用相近的提示拼装与语气约束思路；在深度研究中，上述附录还可并入研究主题描述，并转换为研究管线内部使用的「文档列表」供证据提炼阶段消费。

（截图占位：图4-11 同一会话绑定知识库后，助手回复中出现引用标记或命中统计展示的截图（若前端展示 ragHitCount）。）

4.5  深度研究模块

4.5.1  Planner 子问题规划实现

规划阶段由独立模块调用大模型：构造 Planner 角色提示词，要求其输出结构化的研究计划（含标题、子问题列表与验收要点等），并在提示层约束子问题数量与语义去重；若解析失败则按策略重试。任务状态与过程轨迹写入数据库中的研究任务表，便于进度查询与事后展示。

4.5.2  联网检索与网页抓取/降级实现

联网检索与网页正文抓取、清洗分别在独立模块中实现；研究编排器按子问题并发拉取资料，将知识库命中与网页来源合并为统一的「文档列表」。当会话或策略配置为以知识库为主且命中质量较高时，可跳过外网检索以缩短耗时。

4.5.3  Reader、Writer、Critic 阶段实现

证据提炼、章节写作、质量检查与可选的全文润色分别由独立模块实现，由研究编排器按顺序或并发策略串联；各阶段调用大模型时配合统一的重试与 JSON 容错策略，避免单次解析失败导致整任务中断。

4.5.4  Trace 记录与产物组织

研究任务的状态、中间产物键名与过程轨迹（脱敏后）持久化在同一张任务表中。前端通过「任务详情」类接口轮询状态与轨迹摘要；另提供管线结构接口，将轨迹转换为流程图或阶段节点展示所需的数据，便于用户理解多智能体协作过程。

4.6  前端实现要点

4.6.1  聊天与路由壳层

前端以入口脚本挂载根组件，路由与登录态由壳层组件统一包裹；主界面集中处理会话切换、发送消息、深度研究进度轮询及个性化参数等状态，样式采用全局样式表维护。与后端通信采用浏览器原生请求方式，在请求头中携带令牌以实现鉴权。

4.7  本章小结

本章从服务端接口与数据落盘、对话与知识库检索、深度研究编排及前端交互组织等方面说明了系统的实现方式，并通过截图占位与 PlantUML 源码标出宜配图位置。实现上与第3章设计总体一致：主流程以直连大模型与检索增强为主，并在需要时进入多阶段研究编排；ReAct 工具子系统单独实现以便扩展。后续第5章将在此基础上开展测试与结果说明。
