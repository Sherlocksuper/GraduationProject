二、已完成的研究工作及成果

（一）系统架构与基础能力建设

项目已完成前后端架构搭建与联调。后端基于 Node.js + Express 提供研究任务服务，前端基于 React + Vite 提供可视化交互页面。系统对外提供统一任务接口：创建任务（POST /api/research）、查询状态（GET /api/research/:id）、获取产物（GET /api/research/:id/artifact/:name），能够满足演示、调试与后续评测脚本调用三类需求。

在任务编排方面，系统采用显式状态机管理生命周期，状态覆盖 created、planning、collecting、reading、writing、reviewing、done、failed。每个阶段均有明确输入、输出与异常分支，避免“流程中断后无法追责”的问题。通过状态机分层，系统具备了后续插入新能力（例如信源过滤、冲突分析、持久化策略）的工程空间。

在产物管理方面，系统已稳定输出 plan.json、sources.json、notes.json、report.md、critic_report.json 等关键文件。该产物集合不仅用于页面展示，也可作为论文中的实验样例和结果依据。相较仅输出最终答案，这种“多产物可追溯”设计更符合研究任务对透明性和可验证性的要求。

（二）端到端研究流程实现

系统已实现“主题输入→计划生成→搜索抓取→证据提炼→报告写作→质量检查”的完整闭环：

1）规划阶段（Planner）：根据主题生成子问题和检索关键词，形成可执行计划，降低后续检索噪声。  
2）检索阶段（Researcher）：调用 Tavily 获取结构化检索结果，保留 query、url、title、snippet、provider 等信息。  
3）抓取阶段（Fetcher）：对候选网页进行正文抓取与清洗；当出现反爬或抓取失败时，自动降级使用 snippet。  
4）阅读阶段（Reader）：将原始文本抽取为 claim/support/url 的证据单元，写入 notes。  
5）写作阶段（Writer）：根据证据单元分节生成报告正文，并附来源链接。  
6）审查阶段（Critic）：检查来源充分性、域名多样性与硬断言风险，并在必要时触发有限自动修正。

该流程的核心价值在于“先证据后结论”。与单轮大模型直接写长文相比，本系统将研究任务拆解为可验证步骤，使每个结论都能回溯来源。这种证据驱动写作方式更适合研究场景，也更符合毕业设计对方法严谨性的要求。

（三）可解释性与鲁棒性增强

本课题在中期阶段重点补齐“系统为什么输出这个结果”这一关键能力。当前 trace 机制可记录 reasoning、action、observation、decision、final 等事件类型，并在前端按时间顺序回放。通过该机制，能够完整呈现模型在不同阶段的动作、输入与观察结果，显著降低黑盒感。

在工程鲁棒性方面，已针对高频故障建立基础保障：

1）LLM 调用异常（过载、超时、返回格式不稳定）时，采用指数退避重试；  
2）Reader/Writer 解析失败时，采用降级策略，避免“一处失败全链路失败”；  
3）抓取受限时，自动回退 snippet 证据，保证任务可继续执行；  
4）服务端加入优雅关闭机制，降低热重载下端口冲突影响；  
5）Critic 提供最小质量兜底，避免来源不足或断言过强内容直接输出。

通过上述措施，系统在非理想环境中的稳定性明显提升。当前测试结果表明，即使出现局部抓取失败或模型输出波动，任务仍可产出结构化报告并保留来源链路，满足中期“可运行原型”的要求。

四、存在的问题与困难

当前推进中仍存在以下问题：

1）外部依赖波动：LLM 服务偶发过载、超时或格式异常，会影响任务时延和稳定性。  
2）抓取环境复杂：部分站点反爬强，正文抓取成功率受限，需依赖 snippet 兜底。  
3）信源质量不均：检索结果中仍可能出现低质量页面，影响报告严谨性。  
4）质量评估深度不足：Critic v0 仍属于最小检查，尚未完成强约束交叉验证。  
5）持久化能力待完善：当前以内存任务为主，服务异常重启后上下文可能丢失。

针对以上问题，项目已制定对应缓解策略：通过重试与降级提升可用性，通过信源筛选与 Critic 增强提升可信度，通过持久化和断点恢复提升稳定性。上述策略均已写入第三部分时间计划，具备明确实施节点。

另外，课题还面临“工程推进快、论文沉淀慢”的常见矛盾。为降低该风险，后续将坚持“边开发边记录”原则：关键改动必须配套样例、截图与结论记录，保证每项实现都可沉淀为论文素材。
