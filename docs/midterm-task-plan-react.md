# 中期答辩任务规划（ReAct 多智能体 DeepResearch 原型）

本文档用于**中期答辩**的实现规划与交付对齐：不拘泥于现有代码实现细节，以一个可演示、可解释、可评测的 **ReAct（Reason + Act）多智能体“深度研究”原型**为目标。

---

## 1. 目标（Midterm Objectives）

中期答辩最重要的是：**能跑通、能演示、能解释、能复现**。

- **目标 1：端到端闭环**
  - 输入一个研究主题（开放式问题）
  - 系统自动生成研究计划 → 搜集材料 → 摘要整合 → 生成结构化报告（初稿）
  - 报告包含**引用来源**（URL 或文档片段引用）与可追溯证据

- **目标 2：ReAct 可解释性**
  - 关键步骤以 ReAct trace 形式记录：每一步“行动（tool call）”与“观察（observation）”
  - 可在前端或接口中回放：为何这样搜、用了哪些证据、如何写出结论

- **目标 3：可控与可靠（最小质量保障）**
  - 基本的**信源筛选**与**冲突标记**
  - 基本的**生成-批判-修正**闭环（至少对“引用缺失/事实无证据”能自动修正）

---

## 2. 非目标（Non-goals）

为保证中期可交付，以下不作为必须项（可作为后续展望）：

- **不追求**覆盖所有领域的完美事实正确率（重点是流程、证据、可追溯）
- **不追求**复杂分布式架构的完整落地（先做单机可运行；队列化可做“可选增强”）
- **不追求**大而全的通用 Agent 平台（只为“研究报告生成”场景深度优化）

---

## 3. 总体架构（ReAct + 多智能体协作）

### 3.1 核心思想

- **ReAct**：让模型在“推理（Reason）”与“行动（Act）”之间循环，通过工具获取证据并逐步逼近目标输出。
- **多智能体**：用分工降低单模型在长任务中的不稳定性，并提升可解释性与可控性。

### 3.2 角色分工（建议最小 4 角色）

- **Coordinator（总控/编排器）**
  - 负责把主题拆成阶段性子任务，调度下游智能体，汇总产物并推进状态机
- **Planner（规划智能体）**
  - 把主题转为结构化研究计划（Query Plan / Outline Plan）
  - 输出：子问题列表、搜索关键词、优先级、预期信源类型
- **Researcher（检索智能体）**
  - 调用搜索工具获取链接/文档，并做初步相关性筛选
- **Writer + Critic（写作与批判）**
  - Writer 负责生成段落与整合
  - Critic 负责检查：引用覆盖、事实与证据一致性、是否存在明显幻觉/夸大，并驱动修正

> 中期建议先做“Coordinator + Planner + Researcher + Writer/Critic（可同一模型不同 prompt）”，避免过多 agent 造成工程复杂度暴涨。

---

## 4. Tool 设计（Act 的落点）

中期阶段“工具”不需要很多，但必须能体现 ReAct 的行动价值与证据闭环。

### 4.1 必备工具（中期必须）

- **`web_search`**
  - 输入：`{ query, topK, filters? }`
  - 输出：`[{ title, url, snippet, source, publishedAt? }]`
  - 作用：获取多源候选证据入口

- **`web_fetch_clean`**
  - 输入：`{ url }`
  - 输出：`{ title, url, text, extractedAt, metadata? }`
  - 作用：抓取网页并提取正文（去导航/广告/脚本）

- **`chunk_summarize`**
  - 输入：`{ text, focus, cite: { url } }`
  - 输出：`{ bullets: [{ claim, evidence_quote, confidence }], citations: [...] }`
  - 作用：把长文本转为“可引用要点 + 证据句”

- **`report_render`**
  - 输入：结构化报告（JSON）
  - 输出：`report.md`、`report.txt`
  - 作用：格式化交付

### 4.2 可选增强工具（中期加分项）

- **`domain_rank`**：按域名白名单/黑名单与启发式评分排序
- **`dedupe`**：URL/内容相似去重
- **`fact_backcheck`**：对关键断言再检索一轮（轻量回溯验证）
- **`local_kb_search`**：对本地文档（PDF/Markdown）做检索，用于“资料包研究”演示

## 5. 数据协议与 Trace（答辩展示关键）

### 5.1 任务对象（Research Task）

建议统一一个任务结构（便于 API、前端进度、论文插图）：

- `taskId`
- `topic`
- `status`: `created | planning | collecting | reading | synthesizing | writing | reviewing | done | failed`
- `artifacts`
  - `plan.json`（Query Plan / Outline）
  - `sources.json`（所有来源列表与评分）
  - `notes.json`（摘要要点与引用证据）
  - `report.md` / `report.txt`
- `trace.jsonl`（或 JSON）：完整 ReAct 事件流

### 5.2 ReAct Trace 事件（最小规范）

每条事件至少包含：

- `ts`
- `stage`（如 `planning/collecting/...`）
- `agent`（Planner/Researcher/Writer/Critic/Coordinator）
- `type`: `reasoning | action | observation | decision | final`
- `payload`

展示建议：

- 答辩时展示 1 条完整 trace：从“计划 → 搜索 → 抓取 → 摘要 → 写作 → 批判 → 修正 → 输出”
- 强调“系统不是直接瞎编”，每个关键结论都有来源链接

## 6. 中期最小可演示闭环（MVP Scope）

### 6.1 MVP 功能清单（必须能在现场演示）

- **输入主题 → 生成计划**（可见）
- **执行搜集**（可见进度：正在搜索/抓取第 N 条）
- **生成报告初稿**（Markdown，带引用）
- **一轮 Critic 质量检查**（至少检查“引用覆盖率”和“无来源事实”）
- **导出与复现**（任务可通过 `taskId` 再次打开，trace 可回放）

### 6.2 推荐演示题目（准备 2 个备用）

- 题目 A（偏技术趋势）：例如“未来 3-5 年 Agent 在软件工程中的应用形态与风险”
- 题目 B（偏行业分析）：例如“某一技术在国内外监管与产业落地差异”

选择原则：**可搜到足够资料**、观点有分歧可展示“冲突标记”、能体现“多源引用”。

## 7. 里程碑与实现顺序（Midterm Timeline）

> 这里按“每一步都有可验收产物”的顺序排；你可以按剩余时间把每个里程碑压缩到 1-3 天。

### Milestone 1：任务骨架 + 状态机（Day 1-2）

- 产出：
  - `taskId`、任务状态流转
  - `trace` 落盘（哪怕只记录阶段和耗时）
  - API：create / status / result
- 验收：
  - 任何主题都能创建任务，并能轮询直到完成（先返回占位报告也行）

### Milestone 2：Planner（Day 3-4）

- 产出：
  - Query Plan（子问题 + 关键词 + 优先级）JSON
  - 失败兜底：主题太宽时生成澄清问题（或默认约束）
- 验收：
  - 给定主题，计划稳定、可读、可解释

### Milestone 3：Search + Fetch + Clean（Day 5-7）

- 产出：
  - 搜索结果列表 + 信源评分（最小启发式）
  - 网页正文提取与清洗
- 验收：
  - 对 30-50 个 URL 抓取成功率与耗时可量化；可在答辩展示“来源库”

### Milestone 4：摘要与证据卡片（Day 8-10）

- 产出：
  - 每个子问题：若干“要点（claim）+ 证据句（quote）+ URL”
  - 去重/聚合（最小版：同义合并 + 冲突标记）
- 验收：
  - 任意一条结论都能点击回到来源（可追溯）

### Milestone 5：Writer + Report（Day 11-13）

- 产出：
  - 报告大纲 + 逐节成文
  - 输出 `report.md` / `report.txt`
- 验收：
  - 报告结构清晰、段落可读、引用存在且合理

### Milestone 6：Critic（Day 14-15，加分项但很建议做）

- 产出：
  - 质量检查报告（如“引用覆盖率”“无证据断言列表”“冲突点列表”）
  - 自动修正：对无证据断言触发补搜/删改/降级表述
- 验收：
  - 同一主题在有/无 Critic 的对比下，引用覆盖率明显提升

## 8. 评测与答辩材料（建议提前准备）

### 8.1 最小评测集

- 主题集：至少 10 个（2 个用于现场演示，8 个用于截图/统计）
- 指标（可简单但要可计算）：
  - **引用覆盖率**：报告句子/要点中带引用的比例
  - **来源多样性**：不同域名数、平均每节引用数
  - **冲突标记率**：出现分歧时是否能标注并给出两方来源
  - **端到端耗时**：总耗时 + 各阶段耗时拆分

### 8.2 答辩展示建议（10 分钟版本）

- 1 分钟：问题与目标（信息爆炸、传统搜索/单 LLM 的局限）
- 2 分钟：架构（ReAct 循环 + 多智能体分工 + 工具链）
- 5 分钟：现场演示（topic → plan → sources → notes → report → critic）
- 2 分钟：指标与对比（至少展示引用覆盖/耗时拆分/一次完整 trace 截图）

---

## 9. 风险与预案（现场不翻车）

- **LLM 不稳定/幻觉**
  - 预案：强制“无引用不下结论”；temperature 降低；关键阶段重试；输出校验（JSON schema）
- **搜索/抓取失败（网络波动、反爬、超时）**
  - 预案：准备离线缓存（演示题目提前跑一遍）；抓取失败可退化为只用 snippet 摘要
- **成本/耗时过高**
  - 预案：限制每个子问题 topK、并发上限；缓存；分阶段保存中间产物可断点续跑

---

## 10. 与论文写作对齐（中期就能开始写）

建议论文结构直接对齐上述模块：

- 背景与意义：信息爆炸 + 研究自动化需求
- 相关工作：Agent、ReAct、DeepResearch 类系统
- 系统设计：角色分工、工具协议、状态机、trace
- 关键实现：抓取清洗、证据卡片、写作与批判闭环
- 实验评测：引用覆盖率、来源多样性、耗时拆分、案例分析
- 总结展望：更强的知识库、更多工具、更稳的批判与修正
